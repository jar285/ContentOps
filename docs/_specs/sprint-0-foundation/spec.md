# Sprint 0: Foundation

## Problem Statement
Sprint 0 establishes the foundational repository structure, dependency baseline, and deployment readiness for ContentOps. It provides a statically typed Next.js App Router application backed by SQLite and verified by Vitest and Biome. It does not implement product features; it proves the foundation can build, run, and pass strict quality gates in both local and Vercel environments.

## Invariants
1. **Charter Invariant (Section 4):** All tool schemas and executable logic must derive from a single registry. (Sprint 0 establishes the stack that will support this, though the registry is deferred).
2. **Environment Parity:** The environment schema must validate completely at boot. Any missing required variable must crash the application immediately.
3. **Vercel Build Readiness:** A Vercel-compatible build (`npm run build`) must succeed, producing a deployable artifact.
4. **Role Anticipation:** The system must anticipate the existence of Creator, Editor, and Admin roles, though specific capability mappings are strictly deferred.

## Architecture

### Chosen Libraries
- **Framework:** Next.js `16.x`
- **UI:** React `19.x`
- **Styling:** Tailwind CSS `4.x`. The `src/app/globals.css` file must exclusively use the new `@import "tailwindcss";` directive instead of deprecated v3 `@tailwind` directives.
- **Database:** `better-sqlite3` `12.x`
- **Validation:** `zod` `4.x` (Updated to the current major version 4 for optimal performance and reduced bundle size)
- **Testing:** `vitest` `3.x`
- **Testing DOM:** `@testing-library/react` `16.x`, `@testing-library/dom` `10.x`, `@testing-library/jest-dom` `6.x`, and `happy-dom` `20.x`
- **Linting & Formatting:** `biome` `2.x`
- **Scripting:** `tsx` `4.x` (Dev dependency for executing TypeScript scripts directly)

### Vercel Database Configuration
Vercel serverless functions have an ephemeral, read-only filesystem (except for `/tmp`). To survive deployment, demo-mode treats the database as strictly read-only. The SQLite file is shipped as a build-time asset, and users cannot register on the deployed demo. 

**Runtime Configuration:** 
- In demo mode (`CONTENTOPS_DEMO_MODE=true`), instantiate `better-sqlite3` with `{ readonly: true }` to suppress WAL and journal file creation. In non-demo mode, instantiate writable.
- Any route accessing the database must explicitly declare `export const runtime = 'nodejs'`.

**Build Configuration:**
- `better-sqlite3` relies on a native binary that is not automatically bundled. `next.config.ts` must explicitly declare `serverExternalPackages: ['better-sqlite3']`.
- To guarantee the seeded `.db` file is bundled into the serverless function payload, `next.config.ts` must use `outputFileTracingIncludes` (e.g., `outputFileTracingIncludes: { '/*': ['./data/**/*'] }`).

### Database Schema: Minimal Users Table
The initial users table in `better-sqlite3` must use the following schema:
- `id`: `TEXT PRIMARY KEY`
- `email`: `TEXT UNIQUE NOT NULL`
- `role`: `TEXT NOT NULL CHECK(role IN ('Creator', 'Editor', 'Admin'))`
- `created_at`: `INTEGER NOT NULL DEFAULT (unixepoch())`

The `id` column uses `crypto.randomUUID()` generated by the application (no SQLite AUTOINCREMENT, no external ID library).

### Complete Environment Variable List
- `CONTENTOPS_DB_PATH`: Path to the SQLite database. Defaults to `./data/contentops.db` for local development.
- `CONTENTOPS_DEMO_MODE`: Enables public guardrails. Schema must use `z.coerce.boolean()` to transform the string to a boolean.
- `CONTENTOPS_ANTHROPIC_MODEL`: Model ID for LLM calls (e.g., `claude-haiku-4-5`).
- `CONTENTOPS_DAILY_SPEND_CEILING_USD`: Defines the daily spend limit. Defaults to `2`. Schema must use `z.coerce.number()` to transform the string to a number.
- `ANTHROPIC_API_KEY`: API key for Anthropic. Must be marked as required with absolutely no default value.

### Downstream Implications
The read-only demo decision constrains later sprints: no demo-mode registration (Sprint 2), no user-uploaded corpus on the demo (Sprint 3), mutating tool calls must be simulated or ephemeral on the demo (Sprint 5), and audit log entries on the demo must be seeded rather than user-generated (Sprint 6). This prevents re-discovery later.

## Acceptance Criteria
- The package configuration specifies the pinned versions for Next.js 16, React 19, Tailwind 4, better-sqlite3, Zod 4, Biome, tsx, Vitest, and the `@testing-library` + `happy-dom` suite.
- The `vitest.config.ts` must be configured to discover co-located `.test.tsx` and `.test.ts` files (e.g., `include: ['src/**/*.test.{ts,tsx}']`), use `environment: 'happy-dom'`, and include a setup-files pattern for registering `@testing-library/jest-dom/vitest`.
- The Zod schema strictly validates and coerces the defined environment variables at runtime.
- A basic component integration test renders the `/` page component with Vitest, happy-dom, and Testing Library, and asserts that the placeholder page content is visible.
- The `npm run build` command succeeds completely from a clean checkout, bundling `better-sqlite3` successfully via `serverExternalPackages`.
- The seed script (`src/db/seed.ts`) uses path aliases safely (Context7 verified that `tsx 4.x` resolves `tsconfig.json` paths natively).
- The seed script is strictly idempotent (e.g., via `INSERT OR IGNORE`) to safely execute repeatedly. A single `build` script (`next build`) is maintained locally, while Vercel project settings must explicitly override the build command to `npm run db:seed && npm run build`. This approach (Option A) cleanly separates local stateless builds from Vercel's ephemeral lifecycle requiring a pre-seeded asset.
